Module 01 — Foundations
How Modern LLMs Work
From "bag of words" NLP to Transformers. Understanding the architecture that changed everything.
The Paradigm Shift

Traditional NLP (what you may know) relied on handcrafted feature engineering, rule-based systems, and shallow ML models like Naive Bayes or LSTMs. Modern LLMs are built on a single architecture — the Transformer — and trained on internet-scale text to develop general language understanding.

The key difference: instead of training a separate model for each NLP task (sentiment, NER, translation), an LLM is one giant model that can do all of them via prompting. No task-specific training required.

The Transformer Architecture
✨
Self-Attention
Each token looks at every other token in the sequence and computes a weighted "relevance" score. This is how the model understands context — "bank" means different things near "river" vs "money".
✨
Multi-Head Attention
Multiple attention operations run in parallel, each learning different types of relationships — syntactic, semantic, co-reference. The outputs are combined.
✨
Positional Encoding
Unlike RNNs, Transformers process all tokens in parallel. Positional encodings inject information about token order into the representations.
✨
Feed-Forward Layers
After attention, each token goes through a feed-forward network independently. This is where "facts" and learned knowledge are thought to be stored.
💡
Why this matters for your AI engineering path
You don't need to implement a Transformer. But understanding attention helps you understand why LLMs can parse complex queries like "affordable gaming laptops with good battery life but not bulky" — it weighs all those constraints simultaneously.
Pre-training vs Fine-tuning vs Prompting
Approach What it means When to use it Cost
Pre-training Train a model from scratch on massive text data Never (you're not OpenAI) $millions
Fine-tuning Further train a pre-trained model on domain-specific data When you need specific behavior/style not achievable via prompting $10s—$1000s
Prompting Guide the model's behavior via text instructions at inference time Most use cases, including yours Per-token API cost
RAG Retrieve relevant context and inject it into the prompt When the model needs access to your specific data Storage + retrieval + API
💡
For your e-commerce search
You'll use Prompting (to extract intent from queries) and RAG-like retrieval (to match products). No fine-tuning needed to start.
Key Model Families You'll Encounter
GPT-4 / GPT-4o (OpenAI) Claude 3.5 / Claude 3 (Anthropic) Gemini 1.5 (Google) Llama 3 (Meta, open source) Mistral / Mixtral (open source) Embedding models: text-embedding-3, all-MiniLM

For intent extraction (parsing user queries), any modern model works. For embeddings (vectorizing products), you'll use a dedicated embedding model — different from a chat model.

Module 02 — Foundations
Tokens, Embeddings & Vector Space
The mathematical foundation of modern NLP. How text becomes numbers that capture meaning.
Tokenization

An LLM doesn't read words — it reads tokens. Tokens are chunks of text produced by a tokenizer (usually BPE — Byte Pair Encoding). Common words become single tokens; rare words split into multiple.

Example — Tokenization 
# "laptops below ksh 1000" might tokenize as:
["lap", "tops", " below", " ks", "h", " 1000"]
# ~6 tokens. Why this matters: you're billed per token,
# and context windows are measured in tokens (e.g. 128k tokens).

Why it matters: Your product descriptions and user queries are all tokenized. Long descriptions cost more to process. Context window limits (how much text the LLM can "see" at once) are measured in tokens.

What Are Embeddings?

An embedding is a dense vector (a list of floats, e.g., 1536 numbers) that represents the meaning of a piece of text. Texts with similar meanings have vectors that are close together in this high-dimensional space.

✨
Semantic similarity
"Budget laptop" and "affordable notebook computer" will have very similar embedding vectors — even though no words match. This is the power over keyword search.
✨
Cosine similarity
The standard way to compare embeddings. Measures the angle between two vectors. Returns a score from -1 to 1; higher = more similar. Used to find matching products.
✨
Embedding models
Separate from chat models. Examples: OpenAI's text-embedding-3-small, Cohere embed, or open-source sentence-transformers. These map text → vector.
Practical Code
Python — Generating Embeddings 
from openai import OpenAI
import numpy as np

client = OpenAI()

def embed(text: str) -> list[float]:
    response = client.embeddings.create(
        model="text-embedding-3-small",  # 1536-dim, cheap
        input=text
    )
    return response.data[0].embedding

def cosine_similarity(a, b):
    a, b = np.array(a), np.array(b)
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

# Example
query_vec   = embed("cheap gaming laptop")
product_vec = embed("Budget-friendly notebook for gaming, RTX 3050")

score = cosine_similarity(query_vec, product_vec)
print(f"Similarity: {score:.3f}")  # Will be high (>0.85)
What to Embed in E-Commerce

For product search, you embed your product catalog once (offline) and store the vectors. At query time, you embed the user's query and find the closest product vectors.

💡
What to embed per product
Don't just embed the product name. Embed a rich text combining: name + description + category + key specs. The richer the text, the better the embedding captures the product's meaning. Price and rating are better handled as metadata filters, not embedded.
Python — Product Embedding Strategy
def product_to_text(product: dict) -> str:
    """Convert a product dict into rich text for embedding."""
    return f"""
    {product['name']} — {product['category']}
    {product['description']}
    Key specs: {', '.join(product['specs'])}
    Brand: {product['brand']}
    """.strip()

# Then embed this combined text, NOT just the name
product_vec = embed(product_to_text(product))

Module 03 — Foundations
Prompt Engineering
The craft of reliably extracting structured, useful behavior from LLMs through text instructions.
The Anatomy of a Prompt
✨
System Prompt
Defines the model's role, persona, output format, and constraints. Runs before the user message. This is where you tell it "You are a query parser. Return only JSON."
✨
User Message
The actual input — in your case, the customer's natural language query. e.g. "show me laptops below ksh 50000 with good reviews".
✨
Few-Shot Examples
Showing the model 2-5 examples of input/output pairs dramatically improves reliability. Include examples of edge cases.
✨
Chain-of-Thought
Adding "Think step by step" or showing reasoning in examples improves accuracy on complex tasks. The model's internal reasoning becomes part of the output.
Query Parsing — The Core LLM Job in Your App

Your LLM's primary job is to take a messy natural language query and return a clean, structured JSON object that your search engine can use. Here's how to engineer that prompt:

Python — Intent Extraction Prompt
SYSTEM_PROMPT = """
You are a product search query parser for an e-commerce site.
Extract structured search intent from the user's query.

Return ONLY valid JSON with these fields:
- category: string or null (e.g. "laptop", "phone", "headphones")  
- max_price: number or null (in KSH)
- min_price: number or null
- min_rating: number or null (1-5 scale)
- brands: list of strings or []
- keywords: list of important product attributes
- sort_by: one of ["relevance", "price_asc", "price_desc", "rating"] 

Examples:
User: "cheap laptops under 40000"
Output: {"category": "laptop", "max_price": 40000, "min_price": null,
         "min_rating": null, "brands": [], "keywords": ["budget"],
         "sort_by": "price_asc"}

User: "best samsung phones with good battery"  
Output: {"category": "phone", "max_price": null, "min_price": null,
         "min_rating": 4.0, "brands": ["samsung"],
         "keywords": ["battery life", "long-lasting"],
         "sort_by": "rating"}

Return ONLY the JSON object. No explanation. No markdown.
"""

def parse_query(user_query: str) -> dict:
    response = client.chat.completions.create(
        model="gpt-4o-mini",  # cheap + fast enough for parsing
        messages=[
            {"role": "system", "content": SYSTEM_PROMPT},
            {"role": "user",   "content": user_query}
        ],
        temperature=0,  # deterministic output for parsing tasks
    )
    import json
    return json.loads(response.choices[0].message.content)
Key Prompt Engineering Principles
  • Be explicit about output format. "Return ONLY valid JSON" beats "return JSON". Add "No explanation. No markdown fences."
  • Temperature = 0 for structured tasks. For parsing/extraction, you want deterministic outputs. Use temperature > 0 only for creative tasks.
  • Use few-shot examples for edge cases. Cover ambiguous queries, missing fields, and multi-criteria queries in your examples.
  • Test with adversarial inputs. "give me stuff" — what does your parser return? Design for graceful degradation.
  • Cheaper/smaller models are often fine for extraction. GPT-4o-mini or Claude Haiku handle structured extraction well at 10x lower cost than full models.
💡
Always validate LLM JSON output
Even with a perfect prompt, LLMs occasionally return malformed JSON, especially at high throughput. Always wrap your json.loads() in try/except and have a fallback (e.g., treat as a keyword-only search).

Module 04 — Search & Retrieval
Vector Databases & Semantic Search
How to store millions of embeddings and retrieve the closest ones in milliseconds.
The Problem Vector DBs Solve

You have 10,000 product embeddings (each a 1536-dimensional vector). A user query arrives. You need to find the top-10 most similar products. Computing cosine similarity against all 10,000 products every query works at small scale — but is too slow at scale. Vector databases solve this with Approximate Nearest Neighbor (ANN) indexing algorithms (HNSW, IVF) that find the closest vectors in microseconds.

Vector DB Options
OptionTypeBest forMetadata Filtering
pgvector PostgreSQL extension You already use Postgres. Simplest to start. ✓ Full SQL
Pinecone Managed cloud Production, no infra management ✓ Metadata filters
Weaviate Open source / cloud Hybrid search built-in ✓ GraphQL
Chroma Open source, local Development & prototyping ✓ Basic filters
FAISS Library (Meta) Custom, high performance, no server ◐ Manual
💡
Recommendation for your transition build
Start with pgvector if you're already using PostgreSQL, or Chroma locally for prototyping. Move to Pinecone for production. All have Python clients with similar APIs.
Metadata Filtering — Critical for E-Commerce

Pure vector search returns "semantically similar" products, but can't enforce "under Ksh 50,000" or "rating ≥ 4.0". You solve this with metadata filters — structured attributes stored alongside the vector.

Python — Chroma with Metadata Filtering
import chromadb

client = chromadb.Client()
collection = client.create_collection("products")

# Index products with metadata
collection.add(
    ids=["prod_001", "prod_002"],
    embeddings=[product_vec_1, product_vec_2],
    metadatas=[
        {"price": 35000, "rating": 4.5, "category": "laptop", "brand": "HP"},
        {"price": 72000, "rating": 4.8, "category": "laptop", "brand": "Dell"},
    ]
)

# Query with semantic search AND metadata filters
results = collection.query(
    query_embeddings=[query_vec],
    n_results=10,
    where={
        "$and": [
            {"price":    {"$lte": 50000}},
            {"rating":   {"$gte": 4.0}},
            {"category": {"$eq":  "laptop"}}
        ]
    }
)
Hybrid Search

The best production search combines dense (vector) search with sparse (keyword/BM25) search. Dense search finds semantically similar products; sparse search catches exact keyword matches (e.g., model numbers like "RTX 4070"). The scores are merged via Reciprocal Rank Fusion (RRF).

Module 05 — Search & Retrieval
RAG — Retrieval Augmented Generation
The most widely used LLM application pattern. Connect LLMs to external, up-to-date knowledge.
What is RAG?

LLMs have two problems: their knowledge is frozen at a training cutoff, and they don't know your specific data. RAG solves both by retrieving relevant documents/data at query time and injecting them into the prompt as context. The LLM then generates a response grounded in that retrieved information.

RAG Applied to Product Search

In your e-commerce context, RAG works slightly differently — you're not generating a text answer, you're retrieving products. But the pattern is the same. You can extend it to have the LLM explain why certain products match, or re-rank retrieved candidates.

Python — RAG-style Product Search
def search_products(user_query: str) -> dict:
    # Step 1: Parse query with LLM → structured filters
    parsed = parse_query(user_query)
    
    # Step 2: Embed the query
    query_vec = embed(user_query)
    
    # Step 3: Retrieve top candidates from vector DB
    candidates = vector_search(
        query_vec,
        filters={
            "category":  parsed.get("category"),
            "max_price": parsed.get("max_price"),
            "min_rating": parsed.get("min_rating"),
        },
        top_k=20
    )
    
    # Step 4 (optional): LLM re-ranks / explains results
    context = format_products_for_llm(candidates)
    explanation = llm_rerank(user_query, context)
    
    return {
        "products": candidates[:10],
        "explanation": explanation  # "Showing laptops under Ksh 40k, sorted by rating"
    }
Advanced RAG Concepts
✨
Re-ranking
After retrieving top-20 candidates, use a cross-encoder model or LLM to re-score them relative to the query. More accurate but slower — only do it on the final shortlist.
✨
Query Rewriting
Use an LLM to expand or rewrite the user's query before searching. "laptop for coding" → also search for "developer notebook", "programming computer".
✨
Chunking Strategy
For long product descriptions, you may split them into chunks and embed each separately. Retrieve the most relevant chunk, then fetch the full product.
✨
Iterative RAG
The LLM evaluates retrieved results, then retrieves again if unsatisfied. More powerful but slower — used in agentic systems (covered in Module 07).

Module 06 — Building Apps
LLM APIs & Structured Outputs
Production patterns for getting reliable, structured data from LLMs at scale.
Structured Output Techniques

Getting an LLM to return valid, predictable JSON is one of the most important skills in building LLM apps. Three approaches, from least to most reliable:

  1. Prompt-based: "Return only JSON" in the system prompt. Works most of the time but not guaranteed. Add JSON validation + retry.
  2. JSON Mode: A special API parameter (OpenAI, Anthropic) that forces the model to only output valid JSON. More reliable, still needs schema validation.
  3. Function Calling / Tool Use: Define a typed schema; the model fills in values. Most reliable — the API enforces the schema structure.
Function Calling (the Gold Standard)
Python — Structured Output via Function Calling
import json
from openai import OpenAI

client = OpenAI()

# Define the schema as a "tool" / function
tools = [{
    "type": "function",
    "function": {
        "name": "parse_search_query",
        "description": "Extract structured search intent from a natural language query",
        "parameters": {
            "type": "object",
            "properties": {
                "category":  {"type": ["string", "null"]},
                "max_price": {"type": ["number", "null"]},
                "min_rating": {"type": ["number", "null"], "minimum": 1, "maximum": 5},
                "brands":    {"type": "array", "items": {"type": "string"}},
                "keywords":  {"type": "array", "items": {"type": "string"}},
                "sort_by":   {"type": "string", "enum": ["relevance", "price_asc", "price_desc", "rating"]},
            },
            "required": ["keywords", "sort_by"]
        }
    }
}]

response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "gaming laptops under 60000 with great reviews"}],
    tools=tools,
    tool_choice={"type": "function", "function": {"name": "parse_search_query"}}
)

parsed = json.loads(
    response.choices[0].message.tool_calls[0].function.arguments
)
# {'category': 'laptop', 'max_price': 60000, 'min_rating': 4.0,
#  'brands': [], 'keywords': ['gaming'], 'sort_by': 'rating'}
Pydantic for Validation (with Instructor)

The Instructor library wraps the OpenAI/Anthropic client to automatically parse and validate LLM outputs into Pydantic models — the cleanest pattern for production:

Python — Pydantic + Instructor
import instructor
from pydantic import BaseModel
from typing import Optional, List, Literal
from openai import OpenAI

client = instructor.from_openai(OpenAI())

class SearchQuery(BaseModel):
    category:  Optional[str]   = None
    max_price: Optional[float] = None
    min_price: Optional[float] = None
    min_rating: Optional[float] = None
    brands:    List[str]         = []
    keywords:  List[str]         = []
    sort_by:   Literal["relevance", "price_asc", "price_desc", "rating"] = "relevance"

def parse_query(user_query: str) -> SearchQuery:
    return client.chat.completions.create(
        model="gpt-4o-mini",
        response_model=SearchQuery,  # key parsing line
        messages=[{"role": "user", "content": user_query}],
        temperature=0
    )

result = parse_query("Samsung phones with good camera under 30000")
print(result.category)   # "phone"
print(result.max_price)  # 30000.0
print(result.brands)    # ["samsung"]
💡
Recommended stack for your AI engineering build
Instructor + Pydantic is the cleanest approach. It handles retries, validation, and type-safe output automatically. Install with: pip install instructor pydantic
Important API Concepts
  • Temperature — Controls randomness. 0 = deterministic (use for parsing). 0.7—1.0 = creative generation.
  • Max tokens — Limits response length. For JSON extraction, 200—500 is plenty. Saves cost.
  • System vs User messages — System sets behavior; user is the input. Never put sensitive logic in user messages (prompt injection risk).
  • Streaming — Stream tokens as they're generated for a better UX. Not needed for backend parsing tasks.
  • Rate limits & retries — API calls fail. Use exponential backoff. Libraries like tenacity handle this cleanly.

Module 07 — Building Apps
LLM Agents & Tool Use
When an LLM doesn't just generate text, but takes actions, uses tools, and reasons iteratively.
What is an LLM Agent?

An agent is an LLM that can decide what to do next — it can call tools, observe results, and loop until it has a final answer. Unlike a one-shot prompt that returns text, an agent operates in a reason → act → observe → repeat loop.

For most e-commerce search use cases, you don't need a full agent — a simple pipeline is faster and cheaper. But understanding agents unlocks more powerful features like "find me laptops with the most recent positive reviews" (which might need a web search tool + a database query tool).

The ReAct Pattern

ReAct (Reason + Act) is the foundational agent pattern. The LLM generates a "thought" (reasoning), then an "action" (tool call), observes the result, and repeats:

ReAct Trace — Natural Language
User: "Find me the best reviewed laptop under 45000 that's good for video editing"

Thought: I need to search for laptops under 45000 KSH that support video editing.
         Key specs: RAM ≥ 16GB, dedicated GPU, good display.
Action: search_products(category="laptop", max_price=45000, keywords=["video editing", "GPU", "RAM"])
Observation: Found 12 products. Top results: [HP Envy, Dell Inspiron 15, Lenovo IdeaPad]

Thought: Let me check the reviews for the top 3 to confirm video editing suitability.
Action: get_reviews(product_ids=["HP_Envy", "Dell_Inspiron_15", "Lenovo_IdeaPad"])
Observation: HP Envy: 4.6★ "Great for video editing". Dell: 4.2★ mixed reviews on heat.

Final Answer: The HP Envy is the best match — highly rated for video editing at Ksh 43,500.
Tools Your Search Agent Might Have
✨
search_products
Vector search with metadata filters. The primary tool — takes category, price range, keywords, returns ranked product list.
✨
get_reviews
Fetch and summarize reviews for specific products. The LLM can reason about review sentiment to refine rankings.
✨
compare_products
Given 2—3 products, generate a structured comparison table. Useful for follow-up queries like "what's the difference between these two?"
✨
clarify_query
If the query is too vague, ask the user a clarifying question rather than guessing. "Do you mean laptop or desktop?"
Frameworks (Don't Over-Engineer)
💡
Framework fatigue is real
LangChain, LlamaIndex, AutoGen, CrewAI — these frameworks add abstraction but also complexity. For a focused use case like product search, start with direct API calls and only introduce a framework if you genuinely need its features. Many production LLM apps are just Python functions.

Recommended Build Order
  1. Product data pipeline — Write the script that embeds all your products and stores them in pgvector with metadata (price, rating, category, brand).
  2. Query parser — Build and test the LLM intent extraction with Instructor/Pydantic. Validate on 50+ test queries.
  3. Search endpoint — Wire parser → embedding → vector search → return results. Test end-to-end locally.
  4. Ranking layer — Implement a scoring function that combines semantic similarity score, rating, and review count.
  5. FastAPI wrapper — Expose as a REST API. Connect to your e-commerce frontend.
  6. Evaluation loop — Track which results users click. Use this data to improve embeddings or ranking weights.
Key Pitfalls to Avoid
  • Don't embed prices/ratings — Put them in metadata filters, not the text you embed. Embeddings don't do numeric comparisons well.
  • Don't call the LLM for every search — Cache parsed results for identical queries. Redis with a 1-hour TTL cuts LLM calls by 40—60%.
  • Don't trust LLM output blindly — Always validate. A malformed max_price could cause a division by zero or wrong SQL.
  • Refresh embeddings when products change — Build a pipeline to re-embed and update the vector DB when product descriptions or specs change.
  • Evaluate before deploying — Define an offline test set: 100 queries with expected top-3 products. Measure recall@10 before shipping.
